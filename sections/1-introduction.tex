\section{Introduction}
\label{sec:introduction}

eBPF is widely used for performance-critical kernel extensions: networking~\cite{cilium,katran}, observability~\cite{tracee}, security enforcement~\cite{krsi}, and scheduling~\cite{sched_ext} inside OS kernels~\cite{Linux-ebpf, windows-ebpf}. When loaded into the Linux kernel, each program is verified for safety and JIT-compiled to native code at load time. As eBPF sees increasingly large-scale deployment~\cite{xxx,yyy}, native code quality on these hot paths directly affects end-to-end efficiency in cloud platforms.

However, current eBPF compilation is under-optimized. We find that with the same eBPF bytecode, a user-space LLVM JIT backend~\cite{llvm_bpf_jit} achieves 40\% faster execution and 50\% smaller native code.
For example, the JIT does not exploit hardware-specific instructions or runtime information. Mature runtimes close comparable gaps through extensible compilation frameworks for multiple architectures and profile-guided optimization~\cite{jvm_hotspot,v8_turbofan,llvm}; the eBPF JIT has none, and existing compile-time optimizers~\cite{k2,merlin,epso} that operate before verification cannot exploit deployment-time opportunities.

This is because the in-kernel JIT~\cite{linux_bpf_jit} trades performance for safety: it performs one-to-one instruction translation on a bytecode interface designed to be straightforward to verify, so the verifier and JIT compiler are simple enough to be fully formally verified~\cite{verifyebpfverifier, jitterbug}.

Closing this gap through existing approaches is impractical. Implementing LLVM-level optimization in the kernel JIT would require tens of thousands of lines of safety-critical code that are difficult to maintain, audit, and upstream. Prior compile-time optimizers~\cite{k2,kfuse,merlin,epso} improve code before deployment but miss runtime-guided specialization and do not provide transparency.
Unlike a standalone application, an eBPF extension is embedded into diverse kernel contexts with varying workloads, hardware, and runtime behavior that a compile-time compiler cannot anticipate. We find that even when legal lowerings are added, statically applying them regresses 68\% of real programs.

Our key insight is that eBPF bytecode, designed as a virtual ISA and treated as the final target of compilation, is better understood as a \emph{deployment-time intermediate representation}. Unlike conventional machine code, it combines structured control flow, type information via BTF, and explicit semantics with a mandatory safety validator (the verifier~\cite{verifier}) that can re-check any transformation with the same guarantees as the original load. This means the end of compilation need not be the end of optimization: bytecode can serve as the substrate for deployment-time analysis and rewriting, with the verifier playing the role of an IR validator that enables an \emph{untrusted kernel optimizer}. Unlike native-code patching, bytecode transformations preserve mandatory safety checking; unlike prior pre-load approaches, they operate after deployment with access to runtime context.

This insight leads to an architecture that decouples optimization \emph{safety} from optimization \emph{correctness}. The kernel provides only a minimal syscall for transparent in-place re-verification and replacement; all optimization policy lives in a userspace daemon that need not be trusted for kernel safety. Invalid rewrites are rejected by the verifier and the original program continues unchanged.

To extend safe optimization to hardware-specific idioms that ordinary eBPF cannot represent, we introduce \emph{kinsn}, an extensible instruction mechanism analogous to LLVM intrinsics.  Each kinsn has two realizations: a canonical expansion to ordinary eBPF that the verifier checks, and an architecture-specific native lowering that the JIT emits. The equivalence between each expansion and its native lowering is formally verified, so the safety properties the verifier establishes transfer to the code the JIT actually emits. New kinsn are delivered as trusted kernel modules, requiring no changes to the kernel code.

We propose \tool, an extensible deployment-time eBPF compilation framework. \tool enables two complementary classes of optimization: \emph{hardware specialization}, which uses kinsn to emit efficient platform-specific code, and \emph{runtime specialization}, which inlines stable runtime values and adapts when they change. We evaluate \tool on a corpus of real eBPF programs from production projects. \tool's deployment-adaptive policy eliminates the regression and achieves up to 40\% improvement, with roughly 15\,ms recompilation latency per program and ${\sim}600$ lines of kernel changes.

\noindent\textbf{Contributions.}
This paper makes three contributions:
\begin{itemize}
    \item We characterize the code-generation gap between eBPF's kernel JIT and LLVM backends, and show that optimization profitability is deployment-dependent, with static application of legal lowerings regressing 68\% of real programs.
    \item We treat eBPF as a deployment-time IR, and design \tool, which offloads optimization to untrusted userspace at the bytecode level, using the kernel verifier as safety gate with minimal kernel changes.
    \item Evaluation on production workloads shows that \tool achieves up to 40\% improvement while preserving verifier-backed safety.
\end{itemize}
