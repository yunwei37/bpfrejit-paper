\section{Introduction}
\label{sec:introduction}

eBPF is widely used for performance-critical kernel extensions: networking~\cite{cilium,katran}, observability~\cite{tracee}, security enforcement~\cite{krsi}, and scheduling~\cite{sched_ext} inside OS kernels~\cite{Linux-ebpf, windows-ebpf}. When loaded into the Linux kernel, each program is verified for safety and JIT-compiled to native code at load time. As eBPF sees increasingly large-scale deployment~\cite{xxx,yyy}, native code quality on these hot paths directly affects end-to-end efficiency in cloud platforms.

However, current eBPF compilation is under-optimized. We find that with the same BPF bytecode, a user-space eBPF LLVM JIT backend~\cite{llvm_bpf_jit} achieves 40\% faster execution and 50\% smaller native code.
For example, the JIT does not exploit hardware-specific instructions (e.g., rotate, conditional move, bit-field extract) or runtime information such as branch profiles or cache behavior. Mature runtimes close comparable gaps through extensible compilation frameworks, tiered recompilation, and profile-guided optimization~\cite{jvm_hotspot,v8_turbofan,llvm}; the eBPF JIT has none, and existing compile-time optimizers~\cite{k2,merlin,epso} that operate before verification cannot exploit deployment-time opportunities.

This is because the in-kernel JIT~\cite{linux_bpf_jit} trades performance for safety: it performs one-to-one instruction translation on a bytecode interface designed to be straightforward to verify, so the verifier and JIT compiler are simple enough to be fully formally verified~\cite{verifyebpfverifier, jitterbug}.

Closing this gap through existing approaches is impractical. Implementing LLVM-level optimization in the kernel JIT would require tens of thousands of lines of safety-critical code that are difficult to maintain, audit, and upstream. Compile-time optimizers such as K2~\cite{k2}, KFuse~\cite{kfuse}, Merlin~\cite{merlin}, EPSO~\cite{epso} improve code before load but miss runtime-guided specialization after deployment and do not provide deployment transparency. We find that even when legal lowerings are added, statically applying them regresses 68\% of real programs. 
Unlike a standalone application, an eBPF extension is embedded into diverse kernel contexts with varying workloads, hardware, and runtime behavior that a compile-time compiler cannot anticipate.

Our key insight is that deployment-time optimization of kernel eBPF programs can be safely offloaded to userspace by operating at the \emph{bytecode} level. We observe that eBPF's safety is established at the bytecode level: the existing verifier~\cite{verifier} can re-check any rewritten program with the same guarantees as the original load, and BPF bytecode is structured enough to serve as an optimization IR for sophisticated rewriting passes.
Unlike native-code patching (livepatch), bytecode rewriting preserves mandatory safety checking; unlike pre-load optimization~\cite{k2,merlin,epso}, it operates at deployment time with access to runtime context.

This insight leads to an architecture that decouples optimization \emph{safety} from optimization \emph{correctness}. The kernel provides only a minimal syscall for transparent in-place re-verification and replacement; all optimization policy lives in a userspace daemon that need not be trusted for kernel safety. Invalid rewrites are rejected by the verifier and the original program continues unchanged (fail-safe).

To extend safe optimization to hardware-specific idioms that ordinary BPF cannot represent, we introduce \emph{kinsn}, an extensible instruction mechanism analogous to LLVM intrinsics. BPF bytecode supports sophisticated optimization passes, but like any fixed ISA, it cannot express platform-specific operations. Kinsn extends the bytecode to close this gap. Each kinsn has two realizations: a canonical expansion to ordinary BPF that the verifier checks, and an architecture-specific native lowering that the JIT emits. The equivalence between each expansion and its native lowering is formally verified, so the safety properties the verifier establishes on the expansion transfer to the code the JIT actually emits. New kinsn are delivered as kernel modules, requiring no changes to the verifier or JIT core. Adding new opcodes upstream would require per-architecture kernel patches and verifier modifications for each instruction; kinsn avoids this entirely.

We propose \tool, an extensible deployment-time eBPF compilation framework. \tool enables two complementary classes of optimization: \emph{hardware specialization}, which uses kinsn to emit efficient platform-specific code, and \emph{runtime specialization}, which inlines stable runtime values and adapts when they change. We evaluate \tool on a corpus of real eBPF programs from production projects, \tool's deployment-adaptive policy eliminates the regression and achieves up to 40\% improvement, with roughly 15\,ms recompilation latency per program and ${\sim}600$ lines of code kernel changes.

\noindent\textbf{Contributions.}
This paper makes three contributions:
\begin{itemize}
    \item We show with corpus-scale evidence that optimization profitability for kernel eBPF programs is deployment-dependent: static, always-on peephole application regresses 68\% of real programs, motivating deployment-time, context-adaptive optimization.
    \item We design and implement \tool, which offloads optimization to an untrusted userspace daemon operating at the bytecode level, with the kernel's existing verifier as the safety gate. The kernel provides only mechanism (REJIT syscall and kinsn); the daemon provides all policy. Kernel changes total ${\sim}600$ lines of code.
    \item We demonstrate that \tool's adaptive policy recovers static regressions, improves end-to-end performance on real deployments, and preserves verifier-backed safety, enabling both hardware specialization and runtime specialization through the same framework.
\end{itemize}

%
% Kernel extensions are performance-critical in modern systems. Linux uses eBPF as a kernel extension mechanism for networking~\cite{cilium,katran}, observability~\cite{tracee}, security enforcement~\cite{krsi}, and scheduling~\cite{sched_ext}. Many of these eBPF programs execute on hot paths, and their native code quality directly affects end-to-end efficiency.
%
% However, current eBPF compilation leaves many optimization opportunities unused. Both eBPF verification~\cite{verifier} and JIT compilation~\cite{linux_bpf_jit} run on the kernel's safety-critical loading path, so the JIT must stay simple. As a result, eBPF optimization remains conservative and largely one-shot at load time. Moreover, a load-time-only compilation pipeline cannot exploit opportunities that appear after deployment, such as runtime profiles and stable runtime values.
%
% The root of the problem is that ordinary BPF serves safety well but serves efficiency poorly. Ordinary BPF bytecode gives the verifier a small, target-independent instruction set with explicit semantics. However, efficient native code often depends on hardware-specific idioms that ordinary BPF does not express directly. Closing this mismatch would require the JIT to reconstruct target-specific intent from generic BPF instruction sequences. A simple kernel JIT cannot do that reliably or extensibly. Consequently, the same bytecode interface that enables verifier-backed safety also creates a structural barrier to efficient code generation.
%
% Existing approaches do not fully resolve this mismatch. One option is to make the in-kernel JIT more aggressive, but that would push fast-evolving, workload-dependent optimization policy into the kernel boundary and expand kernel complexity. Another option is to optimize eBPF in userspace before load or at load time, as in prior systems~\cite{k2,kfuse,merlin,epso,epass,bcf}. These efforts improve parts of the pipeline, but they still miss runtime-guided specialization after deployment, and they do not simultaneously provide deployment transparency, verifier-backed safety, and an extensible substrate for hardware-aware code generation.
%
% Our key insight is that closing this gap requires a new bytecode-level abstraction. We introduce \emph{kinsn}, an extensible bytecode layer aligned with hardware idioms. Each kinsn has two realizations. First, it has an equivalent implementation in ordinary BPF that defines its verifier-visible semantics. Second, it has an architecture-specific native lowering that realizes the same semantics efficiently on a target ISA. In this way, kinsn gives the verifier the semantics it needs and gives the JIT the hardware-specific intent it needs, without weakening the BPF safety model.
%
% Based on kinsn, we propose \tool, a post-deployment compilation framework for live eBPF programs. \tool introduces a new compilation boundary after deployment and splits responsibility accordingly. A privileged userspace daemon owns compilation policy: it profiles live programs, constructs rewrites, decides when specialization or hardening is worthwhile, and selects among profitable alternatives. A small kernel component owns mechanism: it recovers live bytecode, re-verifies rewritten candidates, re-JITs them, and replaces the installed native image safely in place. The verifier remains the final safety gate, and the JIT remains small because it realizes kinsn rather than becoming a general-purpose optimizing compiler.
%
% This design enables online adaptation without changing the existing deployment model. \tool can revisit already-loaded programs with runtime feedback and stable runtime values. In-place replacement keeps applications, loaders, and attachment paths unchanged. The existing verification path rejects invalid rewrites before installation, so rewrites fail closed. New optimization and hardening capabilities can therefore evolve in userspace without expanding the trusted kernel boundary.
%
% We evaluate \tool along five dimensions. We measure generated-code quality on microbenchmarks and real eBPF corpora. We measure end-to-end application benefit in representative deployments. We measure online recompilation overhead, disruption, and replacement latency. We measure safety preservation, semantic correctness, and fail-safe rejection. We also measure extensibility by adding new kinsn and rewrite capabilities.
%

%     \item We show that live recompilation improves generated code and end-to-end performance, preserves safety, and keeps kernel complexity small.
% \end{itemize}
