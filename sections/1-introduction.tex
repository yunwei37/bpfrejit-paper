\section{Introduction}
\label{sec:introduction}

eBPF is widely used for performance-critical kernel extensions: networking~\cite{cilium,katran}, observability~\cite{tracee}, security enforcement~\cite{krsi}, and scheduling~\cite{sched_ext} inside OS kernels\cite{Linux-ebpf, windows-ebpf}. When loaded in to Linux kernel, each program is verified for safety and JIT-compiled to native code at load time. Native code quality on these hot paths directly affects end-to-end efficiency in claude platforms.

However, current eBPF compilation is under-optimized. On the same BPF bytecode, a user-space eBPF runtime LLVM JIT backend\cite{llvm_bpf_jit} achieves 40\% faster execution and 50\% smaller native code. For example, the JIT lacks hardwarespecific optimization (e.g. rotate, conditional move, bit-field extract, SIMT insn) that the JIT~\cite{verifier} could lower, and lack runtime information like branch prediction and cache behavior. Mature runtimes close comparable gaps through extensible compilation frameworks, tiered recompilation, and profile-guided optimization~\cite{jvm_hotspot,v8_turbofan,llvm}; the eBPF JIT has none, and its compile time optimization\cite{k2,merlin,epso} before verification cannot exploit post-deployment opportunities.


This is because The in-kernel JIT trade performance for safety:~\cite{linux_bpf_jit} performs one-to-one instruction translation once on a bytecode interface that designed easy to be verified, to keep it simple and even can be full formally verified~\cite{verifyebpfverifier, jitterbug}.

Closing this gap through existing approaches is impractical. Implementing LLVM-level optimization in the kernel JIT would require tens of thousands of lines of safety-critical code. Compile-time optimizers such as K2~\cite{k2}, Merlin~\cite{merlin}, EPSO~\cite{epso}, ePass~\cite{epass}, and BCF~\cite{bcf} improve code before load but miss post-deployment opportunities and do not provide deployment transparency. Compile time staticly applying them statically regresses 68\% of real programs (geometric-mean slowdown 12\%). Unlike entire application, eBPF extension serves diverse workloads inside context whose runtime requirement a static compiler cannot acheive.

Our design rests on three insights.
%
First, eBPF needs what mature runtimes already provide: an \emph{extensible compilation framework} where new optimizations can be deployed as modular passes without modifying core infrastructure---analogous to LLVM's pass pipeline or JVM HotSpot's tiered recompilation, but adapted to the unique constraints of an operating-system kernel.
%
Second, following the microkernel principle of separating mechanism from policy, we split the framework across the kernel boundary. The \emph{kernel provides minimal mechanism}---a syscall interface for in-place recompilation and an extensible instruction substrate (\emph{kinsn}) whose instructions expand to ordinary BPF for verification and lower to efficient native code for execution. A \emph{userspace daemon provides all optimization policy}---a pluggable pass pipeline that decides what, when, and whether to optimize, informed by static analysis, platform capabilities, and runtime profiling. New optimization capabilities require only a new kernel module and a daemon pass update; zero kernel source changes.
%
Third, eBPF's mandatory verifier~\cite{verifier} uniquely enables this separation by decoupling \emph{safety} from \emph{correctness}. Every rewrite submitted through the recompilation interface is re-verified through the same path as \texttt{BPF\_PROG\_LOAD}: if verification fails, nothing changes (fail-safe). Consequently, the optimizer need not reside inside the kernel's trust boundary. A buggy optimization may produce incorrect program behavior, but it cannot compromise kernel safety. This property does not hold in JVM or LLVM, where a JIT or optimizer bug can crash the runtime; it is specific to eBPF's enforced verification model.

Based on these insights we propose \tool, a post-deployment compilation framework for live eBPF programs. \tool comprises three components: (1)~a \emph{REJIT syscall} that recovers live bytecode, re-verifies rewritten candidates, re-JITs them, and atomically replaces the installed native image in place; (2)~\emph{kinsn}, an extensible instruction mechanism delivered through kernel modules, where each kinsn has a canonical BPF expansion for the verifier and an architecture-specific native lowering for the JIT; and (3)~a \emph{userspace daemon} with 15~optimization and hardening passes organized as a pluggable pipeline.

The same framework enables four classes of capabilities. \emph{Hardware specialization} uses kinsn to emit platform-specific idioms (RORX, CMOV, BEXTR, MOVBE on x86; LDP/STP on ARM64). \emph{Runtime specialization} inlines stable map values, propagates constants, and eliminates dead code---a deoptimization model analogous to JVM's speculative optimization with invalidation. \emph{Security hardening} provides dangerous-helper firewalling, live vulnerability patching, and speculation-barrier injection. All three classes operate transparently: existing applications, loaders, and attachment paths are unchanged.

We evaluate \tool along five dimensions: generated-code quality on microbenchmarks and a corpus of real eBPF programs; end-to-end application benefit in representative deployments (Cilium/Katran, Tracee, Tetragon); online recompilation overhead, disruption, and replacement latency; safety preservation through fail-safe semantics and security-oriented case studies; and extensibility through the effort required to add new kinsn and optimization passes.

\noindent\textbf{Contributions.}
This paper makes three contributions:
\begin{itemize}
    \item We identify a structural gap in eBPF's compilation model---the absence of an extensible, runtime-aware compilation framework---and show with corpus-scale evidence that static, always-on peephole optimization is systematically harmful on real workloads.
    \item We design and implement \tool, which separates optimization mechanism (kernel: REJIT syscall and kinsn) from optimization policy (userspace daemon), leveraging eBPF's mandatory verifier to decouple safety from correctness so that the optimizer operates outside the kernel's trust boundary. The kernel changes total roughly 600 lines of code.
    \item We demonstrate that \tool's dynamic policy recovers the regressions caused by static application, improves code quality and end-to-end performance on real deployments, preserves verifier-backed safety, and enables new security-hardening capabilities through the same framework.
\end{itemize}

%
% Kernel extensions are performance-critical in modern systems. Linux uses eBPF as a kernel extension mechanism for networking~\cite{cilium,katran}, observability~\cite{tracee}, security enforcement~\cite{krsi}, and scheduling~\cite{sched_ext}. Many of these eBPF programs execute on hot paths, and their native code quality directly affects end-to-end efficiency.
%
% However, current eBPF compilation leaves many optimization opportunities unused. Both eBPF verification~\cite{verifier} and JIT compilation~\cite{linux_bpf_jit} run on the kernel's safety-critical loading path, so the JIT must stay simple. As a result, eBPF optimization remains conservative and largely one-shot at load time. Moreover, a load-time-only compilation pipeline cannot exploit opportunities that appear after deployment, such as runtime profiles and stable runtime values.
%
% The root of the problem is that ordinary BPF serves safety well but serves efficiency poorly. Ordinary BPF bytecode gives the verifier a small, target-independent instruction set with explicit semantics. However, efficient native code often depends on hardware-specific idioms that ordinary BPF does not express directly. Closing this mismatch would require the JIT to reconstruct target-specific intent from generic BPF instruction sequences. A simple kernel JIT cannot do that reliably or extensibly. Consequently, the same bytecode interface that enables verifier-backed safety also creates a structural barrier to efficient code generation.
%
% Existing approaches do not fully resolve this mismatch. One option is to make the in-kernel JIT more aggressive, but that would push fast-evolving, workload-dependent optimization policy into the kernel boundary and expand kernel complexity. Another option is to optimize eBPF in userspace before load or at load time, as in prior systems~\cite{k2,kfuse,merlin,epso,epass,bcf}. These efforts improve parts of the pipeline, but they still miss runtime-guided specialization after deployment, and they do not simultaneously provide deployment transparency, verifier-backed safety, and an extensible substrate for hardware-aware code generation.
%
% Our key insight is that closing this gap requires a new bytecode-level abstraction. We introduce \emph{kinsn}, an extensible bytecode layer aligned with hardware idioms. Each kinsn has two realizations. First, it has an equivalent implementation in ordinary BPF that defines its verifier-visible semantics. Second, it has an architecture-specific native lowering that realizes the same semantics efficiently on a target ISA. In this way, kinsn gives the verifier the semantics it needs and gives the JIT the hardware-specific intent it needs, without weakening the BPF safety model.
%
% Based on kinsn, we propose \tool, a post-deployment compilation framework for live eBPF programs. \tool introduces a new compilation boundary after deployment and splits responsibility accordingly. A privileged userspace daemon owns compilation policy: it profiles live programs, constructs rewrites, decides when specialization or hardening is worthwhile, and selects among profitable alternatives. A small kernel component owns mechanism: it recovers live bytecode, re-verifies rewritten candidates, re-JITs them, and replaces the installed native image safely in place. The verifier remains the final safety gate, and the JIT remains small because it realizes kinsn rather than becoming a general-purpose optimizing compiler.
%
% This design enables online adaptation without changing the existing deployment model. \tool can revisit already-loaded programs with runtime feedback and stable runtime values. In-place replacement keeps applications, loaders, and attachment paths unchanged. The existing verification path rejects invalid rewrites before installation, so rewrites fail closed. New optimization and hardening capabilities can therefore evolve in userspace without expanding the trusted kernel boundary.
%

We measure online recompilation overhead, disruption, and replacement latency. We measure safety preservation, semantic correctness, and fail-safe rejection. We also measure extensibility by adding new kinsn and rewrite capabilities.
