\section{Introduction}
\label{sec:introduction}

eBPF is widely used for performance-critical kernel extensions: networking~\cite{cilium,katran}, observability~\cite{tracee}, security enforcement~\cite{krsi}, and scheduling~\cite{sched_ext} inside OS kernels~\cite{Linux-ebpf, windows-ebpf}. When loaded into the Linux kernel, each program is verified for safety and JIT-compiled to native code at load time.

However, current eBPF compilation is under-optimized. We find that with the same eBPF bytecode, a user-space LLVM JIT backend~\cite{llvm_bpf_jit} achieves 40\% faster execution and 50\% smaller native code. This is because the in-kernel JIT~\cite{linux_bpf_jit} trades performance for safety: it performs one-to-one instruction translation on a bytecode interface designed to be straightforward to verify, so the verifier and JIT compiler are simple enough to be fully formally verified~\cite{verifyebpfverifier, jitterbug}. It does not exploit hardware-specific instructions or runtime information, and existing compile-time optimizers~\cite{k2,merlin,epso} that operate before verification cannot exploit deployment-time opportunities.

Closing this gap through existing approaches is impractical. Implementing LLVM-level optimization in the kernel JIT would require tens of thousands of lines of safety-critical code that are difficult to maintain, audit, and upstream. Prior compile-time optimizers~\cite{k2,kfuse,merlin,epso} improve code before deployment but miss runtime-guided specialization and do not provide transparency.
Unlike a standalone application, an eBPF extension is a short code fragment embedded without knowledge of its deployment context: the same bytecode may be attached to different hooks, run on different microarchitectures, and face different workload patterns. We find that even when legal lowerings are added, statically applying them regresses 68\% of real programs.

Our key insight is that deployment-time optimization of kernel eBPF programs can be safely offloaded to an untrusted userspace optimizer by treating the \emph{bytecode itself} as an optimization IR. Conventional wisdom treats eBPF bytecode as a final compilation target: programs are optimized by the source-level compiler, verified once at load time, and then JIT-compiled to native code. Optimization is assumed to end before verification begins, because the verifier's complexity budget discourages post-verification rewriting. We observe that this boundary is drawn in the wrong place: eBPF's safety is established entirely at the bytecode level, and the existing verifier~\cite{verifier} can re-check any rewritten program with the same guarantees as the original load. The verifier can therefore be \emph{repurposed} as a post-optimization validator, inverting the traditional compile--verify--run pipeline. Unlike native-code patching, bytecode rewriting preserves mandatory safety checking; unlike prior pre-load approaches, it operates at deployment time---the earliest point at which both the runtime environment and the target hardware are known---enabling optimization along both dimensions.

This insight leads to an architecture that decouples optimization \emph{safety} from optimization \emph{correctness}. The kernel provides only a minimal syscall for transparent in-place re-verification and replacement; all optimization policy lives in an \emph{untrusted} userspace daemon. Invalid rewrites are rejected by the verifier and the original program continues unchanged.

Verifier-as-validator also means that \emph{ISA extensibility can be decoupled from verifier complexity}. For target-specific lowering, any IR needs a way to express hardware idioms that ordinary eBPF cannot represent. However, adding new instructions directly would either bypass verification or require complex, architecture-dependent changes to the verifier. We introduce \emph{kinsn}, an extensible instruction mechanism that resolves this tension. Each kinsn has two realizations: a canonical expansion to ordinary eBPF that the verifier checks, and an architecture-specific native lowering that the JIT emits. The equivalence between each expansion and its native lowering is formally verified once per kinsn, so the safety properties the verifier establishes transfer to the code the JIT actually emits. New kinsn are delivered as trusted kernel modules, requiring no changes to the kernel code.

We propose \tool, an extensible deployment-time eBPF compilation framework. \tool exploits both dimensions of deployment-time information: \emph{hardware specialization}, which uses kinsn to emit efficient platform-specific code, and \emph{runtime specialization}, which inlines stable runtime values and adapts when they change. We evaluate \tool on a corpus of real eBPF programs from production projects. \tool's deployment-adaptive policy eliminates the regression and achieves up to 40\% improvement, with roughly 15\,ms recompilation latency per program and ${\sim}600$ lines of kernel changes.

\noindent\textbf{Contributions.}
This paper makes three contributions:
\begin{itemize}
    \item We characterize the code-generation gap between eBPF's kernel JIT and LLVM backends, and show that optimization profitability is deployment-dependent, with static application of legal lowerings regressing 68\% of real programs.
    \item We introduce \emph{kinsn}, an extensible instruction mechanism that decouples ISA extensibility from verifier complexity.
    \item We design \tool, a deployment-time eBPF compilation framework that combines kinsn-based hardware specialization with runtime specialization in an untrusted userspace optimizer. Evaluation on production workloads shows that \tool achieves up to 40\% improvement while preserving verifier-backed safety.
\end{itemize}
