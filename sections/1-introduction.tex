\section{Introduction}
\label{sec:introduction}

eBPF is widely used for performance-critical kernel extensions: networking~\cite{cilium,katran}, observability~\cite{tracee}, security enforcement~\cite{krsi}, and scheduling~\cite{sched_ext} inside OS kernels~\cite{Linux-ebpf, windows-ebpf}. When loaded into the Linux kernel, each program is verified for safety and JIT-compiled to native code at load time. 

However, current eBPF compilation is under-optimized. We find that with the same eBPF bytecode, a user-space LLVM JIT backend~\cite{llvm_bpf_jit} achieves 40\% faster execution and 50\% smaller native code. For example, the JIT does not exploit hardware-specific instructions or runtime information. Userspace runtimes like JVM, V8~\cite{jvm_hotspot,v8_turbofan,llvm} close comparable gaps through extensible compilation frameworks for multiple architectures and profile-guided optimization. The eBPF JIT has no such framework, and existing compile-time optimizers~\cite{k2,kfuse,merlin,epso} that operate before verification cannot exploit deployment-time opportunities.

This is because the in-kernel JIT~\cite{linux_bpf_jit} explicitly trades performance for safety: it performs one-to-one instruction translation on a bytecode interface designed to be straightforward to verify, so the verifier and JIT compiler are simple enough to be fully formally verified~\cite{verifyebpfverifier, jitterbug}. A bug happend in userspace runtimes where the optimizer is part of the trusted runtime only crash the process, an optimization bug in the kernel JIT could crash the entire system. So, major optimization is assumed to end before verification begins, because the verifier's complexity budget discourages post-verification rewriting.

Closing this gap through existing approaches is impractical. Implementing LLVM-level optimization in the kernel JIT would require tens of thousands of lines of safety-critical code that are difficult to maintain, audit, and upstream. Unlike a standalone application, an eBPF extension is a small code snippet embedded into diverse kernel contexts with varying workloads, hardware, and runtime behavior that a compile-time compiler cannot anticipate. We find that even when legal lowerings are added, statically applying them regresses 68\% of real programs: optimization profitability is a \emph{deployment} property, not a program property. The same conditional-move lowering, for instance, speeds up one program by 19\% but slows another by 67\%, depending on branch behavior in its deployment context.

Our key insight is that deployment-time optimization of kernel eBPF programs can be safely offloaded to an untrusted userspace optimizer by using the \emph{bytecode} as a verifiable optimization IR. 
Safety should derive from the verifier checking \emph{bytecode}, not from restricting the optimization complexity.
eBPF bytecode is structured enough to serve for optimization passes. The existing verifier~\cite{verifier} can therefore re-check any rewritten program with the same guarantees. This lets the verifier be \emph{repurposed} as a post-optimization validator, inverting the traditional compile--verify--run pipeline.
Unlike native-code patching, bytecode rewriting preserves mandatory safety checking; unlike prior pre-load approaches, it operates at deployment time with access to runtime context.

This insight leads to an architecture that decouples optimization \emph{safety} from optimization \emph{correctness}. The kernel provides only a minimal syscall for transparent in-place re-verification and replacement; all optimization policy lives in an \emph{untrusted} userspace daemon. Invalid rewrites are rejected by the verifier and the original program continues unchanged. This trust model does not require proof construction\cite{necula1997pcc}, and optimization correctness can be further partially validated by comparing the verifier's abstract state before and after rewriting.

To enable extending bytecode expressiveness, \emph{ISA extensibility must be decoupled from verifier complexity}. For target-specific lowering, any IR needs a way to express hardware idioms that ordinary eBPF cannot represent. However, adding new instructions directly would either bypass verification or require complex, architecture-dependent changes to the verifier. We introduce \emph{kinsn}, an extensible instruction mechanism that resolves this tension. Each kinsn has two realizations: a canonical expansion to ordinary eBPF that the verifier checks, and an architecture-specific native lowering that the JIT emits. The equivalence between each expansion and its native lowering is formally verified once per kinsn, so the safety properties the verifier establishes transfer to the code the JIT actually emits. New kinsn are delivered as trusted kernel modules, requiring no changes to the kernel code.

We propose \tool, an extensible deployment-time eBPF compilation framework. \tool exploits both dimensions of deployment-time information: \emph{hardware specialization}, which uses kinsn to emit efficient platform-specific code, and \emph{runtime specialization}, which inlines stable runtime values and adapts when they change. We evaluate \tool on a corpus of real eBPF programs from production projects. \tool's deployment-adaptive policy eliminates the regression and achieves up to 40\% improvement, with roughly 15\,ms recompilation latency per program and ${\sim}600$ lines of kernel changes.

\noindent\textbf{Contributions.}
This paper makes three contributions:
\begin{itemize}
    \item We characterize the code-generation gap between eBPF's kernel JIT and LLVM backends, and show that optimization profitability is deployment-dependent, with static application of legal lowerings regressing 68\% of real programs.
    \item We introduce \emph{kinsn}, an extensible instruction mechanism that decouples ISA extensibility from verifier complexity.
    \item We design \tool, a deployment-time eBPF compilation framework that combines kinsn-based hardware specialization with runtime specialization in an untrusted userspace optimizer. Evaluation on production workloads shows that \tool achieves up to 40\% improvement while preserving verifier-backed safety.
\end{itemize}
