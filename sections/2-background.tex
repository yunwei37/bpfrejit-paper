\subsection{eBPF}
\label{subsec:ebpf}

% 1 BPF Instruction Set Architecture (ISA) — https://docs.kernel.org/bpf/standardization/instruction-set.html
% eBPF Syscall — https://docs.kernel.org/userspace-api/ebpf/syscall.html
% eBPF verifier — https://docs.kernel.org/bpf/verifier.html
% Documentation for /proc/sys/net/ — https://docs.kernel.org/admin-guide/sysctl/net.html
eBPF is Linux's mechanism for running user-defined programs at kernel hooks under verifier-enforced restrictions. Figure~\ref{fig:ebpf} shows the standard path from userspace compilation to in-kernel execution. A program is first compiled in userspace into eBPF bytecode and then submitted through \texttt{bpf(BPF\_PROG\_LOAD)} syscall. The kernel verifier checks the submitted bytecode before installation. It validates the control-flow graph, simulates all execution paths, and tracks register and stack state to ensure safe memory access and valid use of kernel interfaces~\cite{linuxkernel_bpf_verifier}. After a program is accepted, it is attached to the hook defined by its program and attach types. When JIT is enabled, the kernel then translates the accepted bytecode into native instructions before execution~\cite{ebpf_jit}. Later packets or kernel events invoke that installed program body.

\subsection{Optimization Pipelines in Modern Systems}
\label{subsec:modern-pipelines}

Modern runtimes and compiler frameworks both stage validation before optimization. Managed runtimes place validation at program admission, whereas compiler frameworks place verification inside the compilation pipeline. The JVM verifies class files during linking and then uses profiling-guided tiered compilation to optimize hot methods at runtime~\cite{jvm_hotspot}. A WebAssembly runtime first validates a module and then compiles the validated module through an engine-specific backend~\cite{wasmtime}. LLVM uses the compiler-oriented form of the same pattern~\cite{llvm}. LLVM checks whether IR is well formed and then runs analysis and transformation passes through a modular optimization pipeline, including JIT-time transform layers in ORC.

JVM and WebAssembly runtimes use validation as an admission check on programs entering a managed execution environment, whereas LLVM uses verification as an internal consistency check within the compiler pipeline. In the JVM and in WebAssembly runtimes, optimization remains part of the runtime implementation after code admission. In LLVM, extensibility centers on transformation passes and JIT layers, while verification remains a check on IR structure rather than a gate on generated code.
